# Canonical jurisdiction decoupling — model + migration

## Problem

The canonical schema makes `jurisdiction_id` a NOT NULL, identity-defining field across the identity, vote, bill, and session clusters, but those entities relate to a jurisdiction only *through a public Organization* (or not at all, for people). This mis-models the domain and blocks step-6c PM sync (PM person/org reads carry no jurisdiction; org jurisdiction is optional). Design: [`docs/specs/2026-06-09-canonical-jurisdiction-decoupling-design.md`](../specs/2026-06-09-canonical-jurisdiction-decoupling-design.md) (approved).

## Approach

Decouple `jurisdiction_id` from 24 empty tables in four model files, deriving jurisdiction transitively via the public Organization each entity hangs off. `organizations.jurisdiction_id` becomes nullable (the binding root); `persons` and the role/assignment/event, vote, and bill-instance tables drop it; `legislative_sessions` drops it and gains a **required** `organization_id` FK to the WA Legislature org. Bill vocab (`bill_types`, `bill_relationship_types`) and all `statute_*` keep their jurisdiction key; the PDC cluster is untouched. All natural keys lose the `jurisdiction_id` component (collapsing to `(source, source_id)` plus table-specific secondaries). One autogenerated-then-hand-verified Alembic revision applies it; tables are empty so there is no data migration. TDD throughout: tests red against the current NOT NULL model first.

## Tradeoffs / alternatives

- **Keep a nullable denormalized `jurisdiction_id` on derived tables** — rejected: redundant with the org and can drift; the cohort queries are simple joins (decided in design).
- **Phase the migration per cluster (several revisions)** — rejected: tables are empty and the change is mechanical; one revision is simpler to review and apply atomically.
- **Make `legislative_sessions.organization_id` nullable** — rejected: the user requires it (a session always belongs to the legislature org); NOT NULL enforces the invariant. Acceptable because tables are empty (no backfill).
- **Hand-write the migration** — rejected as default: autogenerate gets the bulk right; we hand-verify ordering and the session column add rather than write 24 tables by hand.

## Steps

1. **`identity.py`** — `organizations.jurisdiction_id` → `nullable=True` (keep FK + index); drop `jurisdiction_id` (column + FK + index) from `persons`, `roles`, `assignments`, `entity_events`, `person_identifiers`, `organization_identifiers`; rewrite their natural keys (`uq_*_natural_key` → `(source, source_id)`; roles add `(organization_id, name)`; identifiers → `(scheme, value)` + keep `(parent_id, scheme)`). Verify: module imports; only `organizations` retains `jurisdiction_id` (now nullable).

2. **`votes.py`** — drop `jurisdiction_id` from `vote_events`, `vote_counts`, `person_votes`; natural keys → `(source, source_id)` (keep `(vote_event_id, count_type)` and the person-or-name check). Verify: no `jurisdiction_id` remains in the file.

3. **`bills.py`** — drop `jurisdiction_id` from the 13 instance tables (`bills`, `bill_sponsorships`, `bill_actions`, `bill_action_classifications`, `bill_versions`, `bill_titles`, `amendments`, `bill_subjects`, `bill_relationships`, `bill_events`, `bill_version_links`, `bill_statutory_citations`, `bill_supplements`) and strip `jurisdiction_id` from each of their unique constraints; **leave `bill_types` and `bill_relationship_types` unchanged**. Verify: exactly two tables in `bills.py` still declare `jurisdiction_id`.

4. **`sessions.py`** — drop `jurisdiction_id` from `legislative_sessions`; add `organization_id: Mapped[_ULID]` NOT NULL FK → `{SCHEMA}.organizations` (RESTRICT) + index; natural keys → `(source, source_id)` and `(organization_id, slug)`. Verify: import succeeds; `organization_id` is NOT NULL.

5. **Tests (TDD red→green)** — update `test_identity_pm_anchors.py` to construct entities without `jurisdiction_id` (orgs optionally with it); add tests: person persists with no jurisdiction; org persists with `jurisdiction_id=None` and with a value; `(source, source_id)` uniqueness; a `legislative_session` requires `organization_id`; a role's jurisdiction is reachable via an `organizations` join. Run the relevant test file: confirm red before steps 1–4 land, green after.

6. **Migration** — `uv run alembic revision --autogenerate -m "decouple jurisdiction from identity/vote/bill/session clusters"`; hand-verify: per-table drop-constraint → create-new-constraint → drop column/FK/index ordering; `organizations.jurisdiction_id` altered to nullable (not dropped); `legislative_sessions.organization_id` added NOT NULL with FK + index; bill vocab + statutes untouched. `uv run alembic upgrade head` on the dev DB; spot-check with `\d canonical.persons` / `\d canonical.legislative_sessions`; confirm `downgrade` runs (structural-only).

7. **Verify + ship** — `uv run alembic upgrade head` on the **test** DB; full `uv run pytest` (incl. `-m integration`) green; `uv run ruff check .` clean; commit per step; run `alembic upgrade head` on the production DB (schema-only; live API + sidecar untouched). Update todos; 6c descriptors resume after this + PM #193/#194.

## Open questions / risks

- **`org_type` vocab for the WA Legislature** — no `legislature` value today; add one or use `government_agency`. Settled at *seed* time (the legislature org isn't created by this migration), so it does not block these steps. Noted for the legislature adapter/seed.
- **Seeding order** — once populated, a `legislative_session` requires its legislature org to exist first; the adapter/seed must create the org before sessions. No issue while empty.
- **Autogenerate fidelity for constraint renames** — Alembic may emit drop+create for renamed unique constraints; acceptable (empty tables), but verify no constraint is silently dropped without a replacement.
- **PDC/lobbying cluster** — same pattern, deferred to the PDC adapter workstream (out of scope here).
- **Downgrade** — structural-only (empty tables); re-imposing NOT NULL on a repopulated table would need a default. Documented limitation.
